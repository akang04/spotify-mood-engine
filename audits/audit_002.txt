Fix 1: Popularity removed from feature matrix
  Files: app/cluster.py, app/recommend.py

  cluster.py — build_feature_matrix():
    - Removed popularity column concatenation (was np.hstack([genre_matrix, popularity]))
    - Removed "popularity" from feature_names (was top_genres + ["popularity"])
    - Added ValueError guard when no genre data present — raises with a clear message
      rather than silently returning a zero-column matrix that would crash StandardScaler
    - Feature matrix is now genre one-hot only; return value is (X_scaled, scaler, top_genres)

  cluster.py — save_clusters():
    - Removed genre_names filter (was [f for f in feature_names if f != "popularity"])
    - Simplified centroid loop: all feature_names are now genres, no special-casing needed
    - Updated docstring

  recommend.py — get_library_tracks_for_cluster():
    - Replaced sort_values("popularity", ascending=False) with
      DataFrame.sample(frac=1) random shuffle
    - Popularity was unreliable as a ranking signal (often 0 for unsynced tracks)

  Note: popularity column stays in the DB schema (Track.popularity) and
  load_tracks_dataframe() still returns it — just not used as a clustering feature
  or playlist-ranking signal.

  Files: app/recommend.py, app/ui.py

  ui.py — render_playlist_section():

    - Removed catalog expansion st.checkbox entirely
    - Updated build_playlist() call (removed sp, expand args)

Fix 3: Dead schema weight removed
  File: app/models.py

  - AudioFeatures class removed entirely (was never written to anywhere in the codebase)
  - audio_features relationship removed from Track
  - Centroid audio columns removed from Cluster:
      centroid_valence, centroid_energy, centroid_danceability, centroid_tempo,
      centroid_acousticness, centroid_instrumentalness, centroid_speechiness

  - Float removed from sqlalchemy imports (no longer needed after AudioFeatures removal)

  Existing databases: the audio_features table and extra centroid columns remain as
  inert data in existing SQLite files — SQLAlchemy create_all is additive-only, so
  no migration is needed and no data is at risk. database.py is unchanged.
