====================================================
ISSUES FOUND
====================================================

ISSUE 1 — HTTP for Last.fm API (fetch.py:27)
  _LASTFM_BASE_URL = "http://ws.audioscrobbler.com/2.0/"
  The api_key parameter is sent as a plain-text query string over HTTP.
  Fix: change to "https://ws.audioscrobbler.com/2.0/".
  Severity: low (api_key is read-only, no personal data sent); still worth
  fixing as a security hygiene measure.

ISSUE 2 — genre_distribution ignores lastfm_tags (cluster.py:266-269)
  The "Top Genres per Cluster" bar chart sources genre strings from
  df["genres"] only (Spotify-sourced). The clustering feature matrix uses
  the merged genres + lastfm_tags set. A user who synced with Last.fm enabled
  will see their clusters driven by Last.fm tags but the bar chart will not
  reflect them.
  Fix: in genre_distribution(), iterate both "genres" and "lastfm_tags"
  columns and merge them (same split+strip logic as build_feature_matrix).
  Severity: cosmetic but misleading — bar chart will undercount / mismatch
  cluster character when Last.fm tags are the dominant signal.

ISSUE 3 — Dead session state: scaler (ui.py:116)
  st.session_state.scaler is assigned in render_sidebar() after re-clustering
  but is never read anywhere (render_charts creates its own scaler internally).
  Fix: remove the assignment and the "scaler" key from _SESSION_KEYS.
  Severity: cosmetic / minor memory waste.

ISSUE 4 — Double build_feature_matrix call in render_charts (ui.py:150)
  Every page render calls build_feature_matrix(df) to get X_scaled for PCA,
  independent of the matrix computed during clustering. For large libraries
  this adds latency on every Streamlit re-render (StandardScaler refit +
  tag vocabulary rebuild).
  Fix options:
    a. Cache X_scaled in st.session_state alongside df_clustered (set during
       run_clustering, invalidated on sync or re-cluster).
    b. Use @st.cache_data with df as input.
  Option (a) is simpler and avoids Streamlit cache invalidation complexity.
  Severity: performance — noticeable only for libraries > a few hundred tracks.
